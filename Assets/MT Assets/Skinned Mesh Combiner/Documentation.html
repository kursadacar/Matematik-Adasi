What is the Skinned Mesh Combiner
The Skinned Mesh Combiner is a tool that will help you merge the meshes that use the Skinned Mesh Renderer. The Skinned Mesh Combiner only works with this type of mesh. The Skinned Mesh Combiner has some merging methods, each method has its positive and negative sides, you can see this below in the documentation. See the video demo below and follow the reduction of draw calls with each type of merge.
First steps to combine
Before you begin to merge your meshes, you must first prepare the entire GameObjects hierarchy and properly validate Skinned Mesh Renderers.

Step 1 - Your hierarchy of Skinned Meshes should be as shown in the image below. For example, if you want to combine a character, your meshes must be organized the way you explain the image below...

"Character GameObject" must have the Animator that will be responsible for executing the animations of its Skinned Meshes. The Skinned Mesh Combiner should be in the Root GameObject (Character), because when you start the merge, the Skinned Mesh Combiner will search for all the Skinned Mesh affiliated with it. Each GameObject "SkinnedMeshX" image contains a Skinned Mesh Renderer.
Step 2 - Make sure all of your meshes have materials associated with them. Meshes that have empty materials will be ignored during the merge.

Avoid leaving null materials on Skinned Mesh Renderers. This also helps with optimization.
Step 3 - Prevent your Skinned Mesh Renderers from having null meshes, as any Skinned Mesh Renderer that has null meshes will be ignored during the merge process.

Step 4 - Make sure that all the materials that your meshes use have Main Textures. This is not an obligation, and depends on the merging mode, but it may be good to have textures associated with all materials.

Step 5 - Now that everything is ready, it's just you to combine the meshes! For this, keep reading.
How to combine meshes (In Editor)
To combine meshes using the editor, it's very simple and easy! Let's go step by step. With this method, you will not need to use any line of code, and in addition, when the scene starts your meshes will already be combined!

Step 1 - Add the "Skinned Mesh Combiner" component to the root GameObject of your skinned meshes.

Select the root GameObject for your model. Then click "Add Component." Search for "Skinned Mesh Combiner". Then just add the component that appears in the image above.
Step 2 - Select the Merge Method. Each merge method has advantages and disadvantages, and each one meshes the meshes in a different way. We'll talk about the merge methods below in the documentation.


Just click on the "Merge Method" box and choose one of the options. ( Some merge methods will provide you with a variable called "Material To Use". You must provide a material to be used by the mesh resulting from the merge. Skinned Mesh Combiner will apply the material you supply, that is, a copy of the material you supply, and will apply it to the combined mesh.)
Step 3 - Click the "Combine Meshes!" Button.

Notice: The merge process varies according to the method chosen, the number of vertices that the meshes have, and the size of their textures.
Step 4 - Wait a few moments and when the "Combine Meshes!" switching to "Undo Merge" means that the meshes were combined! After combining, new meshes will be generated and old meshes will be deactivated. You can always disable the merge whenever you want!

Automatically, the Skinned Mesh Combiner will also show you the "Logs Of Merge" tab that contains all messages generated by the Skinned Mesh Combiner while the merge was being performed. This is how the Skinned Mesh Combiner communicates with you, in case you find something wrong, or think it's important to let you know!

- This was the basics for you to combine the stitches of your models! Continue reading the documentation to understand more about all the features of the Skinned Mesh Combiner!
How to combine meshes (At Runtime)
It's also easy to combine meshes while your game runs! However, avoid combining meshes repeatedly in a short time, this can lead to a high memory expense, and depending on the merge method, can cause freezes in your game. Then use this function with caution. Currently, the best method of merge at runtime is the "One Mesh Per Material". You can read about all the methods below in the documentation. Then follow the steps below to merge at run time.

Step 1 - First, import the "MTAssets" library into your script, because without it you will not be able to access the classes of this tool.
Script C#


using MTAssets.SkinnedMeshCombiner;
Step 2 - Find the root GameObject that has the "Skinned Mesh Combiner" component. Then get the "Skinned Mesh Combiner" component. See an example below.
Script C#


void Start()
{
    SkinnedMeshCombiner combiner = RootGameObject.GetComponent<SkinnedMeshCombiner>();
}
Step 3 - Then call the "CombineMeshes()" method of the Skinned Mesh Combiner. When you call this method, the merge will start instantly. The merge method to be used will be what is defined in the Skinned Mesh Combiner component at the time of calling this function.
Script C#


void Start()
{
    SkinnedMeshCombiner combiner = RootGameObject.GetComponent<SkinnedMeshCombiner>();
    combiner.CombineMeshes();
}
- If you want to see if your meshes are currently merged, see the method "isMeshesCombined()".

void Start()
{
    if(combiner.isMeshesCombined() == true)
    {
        //Do something
    }
}
- If you want to undo the merge, just call the "UndoCombineMeshes(true, true)" function.

Script C#


    using System.Collections;
    using System.Collections.Generic;
    using UnityEngine;
    using MTAssets.SkinnedMeshCombiner;
    
    public class demo : MonoBehaviour {
    
        //Public variables
        public GameObject characterGameObject;
        public SkinnedMeshCombiner combiner;
        public Material materialToUseInMergedMeshes;
    
        //To ignore during merge
        public GameObject aGameObjecToIgnore1;
        public GameObject aGameObjecToIgnore2;
    
        void Start () {
            //Find the combiner
            combiner = characterGameObject.GetComponent<SkinnedMeshCombiner>();
    
            //Start merging all meshes
            MergeAllMeshes();
        }
    
        void MergeAllMeshes()
        {
            //Configure all variables
            combiner.mergeMethod = SkinnedMeshCombiner.MergeMethod.AllInOne;
    
            combiner.allInOneParams.materialToUse = materialToUseInMergedMeshes;
            combiner.allInOneParams.atlasResolution = SkinnedMeshCombiner.AtlasSize.Pixels2048x2048;
            combiner.allInOneParams.mergeOnlyEqualsRootBones = false;
            combiner.nameOfThisMerge = "Amazing Merge!";
            //You can change the value of any variable you want, the Skinned Mesh Combiner, much more than that!...
            //combiner.onemeshpermaterial for variables of One Mesh Per Material merge method
            //combiner.allInOneParams for variables of All In One merge method
            //combiner.justMaterialColors for variables of Just Material Colors merge method
            //combiner.onlyAnima2dMeshes for variables of Only Anima2D Meshes merge method
    
            //add the gameobjects to ignore
            if (combiner.gameObjectsToIgnore == null)
                combiner.gameObjectsToIgnore = new List<GameObject>();
            combiner.gameObjectsToIgnore.Add(aGameObjecToIgnore1);
            combiner.gameObjectsToIgnore.Add(aGameObjecToIgnore2);
    
            //Add the on done merge events
            combiner.onCombineMeshs.AddListener(OnDoneMerge);
            combiner.onUndoCombineMeshs.AddListener(OnUndoMerge);
    
            combiner.launchConsoleLogs = true;
    
            //Start the merge
            combiner.CombineMeshes();
        }
    
        void OnDoneMerge()
        {
            Debug.Log("Meshes merged!");
        }
    
        void OnUndoMerge()
        {
            Debug.Log("Meshes not merged! :(");
        }
    
        void UndoTheMergeOfAllMeshes()
        {
            //Undo the merge
            combiner.UndoCombineMeshes(true, true);
        }
    }
How this method works (One Mesh Per Material Merge Method)
Briefly speaking, the "One Mesh Per Material" method combines all meshes that share the same material into one. So if you have 5 meshes that use material A, those 5 meshes will be combined in 1 mesh, which will use this material A. I will explain more precisely below.

First the Skinned Mesh Combiner looks for the Skinned Meshes that are the childrens of the GameObject in which it is. Then, the Skinned Mesh Combiner groups meshes that use the same material. When you have a mesh that uses more than 1 material, the Skinned Mesh Combiner will separate the faces of this mesh according to each material.

Each box represents a GameObject containing the Skinned Mesh Renderer. Green boxes use material A, blue boxes use material B and brown boxes use material C.
After separating each mesh, it creates a new GameObject. This new GameObject will group the new meshes resulting from the merge.

Then after that, the Skinned Mesh Combiner will combine all the meshes and submeshes that share the same material, and insert all into a one mesh inside the GameObject created in past step.

And finally, the Skinned Mesh Combiner simply disables the old meshes.

Pros Of "One Mesh Per Material" Merge Method
  • It is the most optimized and fastest merge method
  • Does not modify any texture, material or UV
  • Can combine meshes that use any type of materials
  • Does not generate too much memory garbage
  • It is the simplest method of merge
  • The lower the amount of materials, the greater the reduction of draw calls
  • Can be called more often than the "All In One" method
  • Fully compatible with mipmaps
  • Keeps animations intact
  • Can be used while Animator performs animations
  • You do not need "Read/Write enabled" permissions in the textures import settings
  • It is compatible with all types of meshes!
  • It is compatible with Normal Maps, Height Maps and other types of effects.
Cons Of "One Mesh Per Material" Merge Method
  • Does not always reduce draw calls dramatically
How this method works (All In One Merge Method)
Briefly speaking, the "All In One" method will merge all meshes into one, and only one new mesh will be generated that will be the junction of all the others. Then it will merge all the textures into one atlas, and will insert that atlas into a new material. At the end, this material will be inserted into the unique mesh. With this merge method, all the textures to be merged must have the "Read/Write enabled" option in the import settings. If you have 5 meshes, only 1 mesh, 1 material and 1 atlas will remain after the merge. This method is a bit more complex, so try to understand well how it works before using it. In addition, this method is not compatible with any mesh.

Some meshes can offer problems for this method, according to the way in which they were created. For example, some meshes may contain errors in your UV map, which can generate errors during the merge, however, the vast majority of the tested meshes did not result in errors during the merge and everything worked perfectly in tests!

First, it merges all the meshes together, creating a new mesh that is the junction of all. However, this new mesh does not have any material.

After you provide a custom material (Material To Use), the Skinned Mesh Combiner will copy your material when creating the new material. This new created material will be inserted into the created mesh.

After this, the Skinned Mesh Combiner will extract the texture of each old mesh and will create an atlas texture. It will create atlas textures according to your preferences, being able to create atlas with textures, normal maps, occlusion maps and etc.

Then the created mesh will have its UVs modified so that it can receive the new atlas. Then the atlas is inserted into the new mesh created.

Finally, old meshes are disabled. Resting only the mesh created with its material and texture with atlas.

Pros Of "All In One" Merge Method
  • Most of the time, it dramatically reduces the amount of draw calls
  • Always reduce calls to 1 draw call, regardless of how many meshes you have
  • Keeps animations intact
  • Can be used while Animator performs animations
  • When you combine compatible meshes with this method, it returns optimum performance
  • It can work with effects like Normal Maps, Height Maps and the like.
Cons Of "All In One" Merge Method
  • It has a longer processing time, depending on the size of the textures, amount of meshes and vertices.
  • Modifies UVs, materials and textures
  • Meshes that have the UV map larger than the texture (to repeat the texture in the mesh) may have the lowest texture
  • It can generate a higher amount of garbage if compared to the "One Mesh Per Material"
  • It can not be called as much as the "One Mesh Per Material" if you want your game to be fast
  • You need to enable the "Read/Write enabled" option for all model textures in the import settings.
  • Always be tested so you know if you will get a desired result
How this method works (Just Materials Colors Merge Method)
The "Just Material Colors" merge method only merges the colors of the materials in your meshes. To do this, it collects the color of each material that its meshes use, creates an atlas that contains a palette of colors for each mesh that was mixed, then color that mesh with its respective color. It is a very simple merge method and made for people who need to merge their models that do not use textures, just material colors!

First, it merges all the meshes together, creating a new mesh that is the junction of all. However, this new mesh does not have any material.

Then a new material is created from the material you provide (Material To Use). This new created material will be inserted into the created mesh.

After this, the Skinned Mesh Combiner extracts the color of each material that its meshes use, and inserts those colors into an atlas.

Then the created mesh will have its UVs modified so that it can receive the new atlas. Then the atlas is inserted into the new mesh created.

Finally, old meshes are disabled. Resting only the mesh created with its material and texture with atlas.

Pros Of "Just Material Colors" Merge Method
  • Most of the time, it dramatically reduces the amount of draw calls
  • Always reduce calls to 1 draw call, regardless of how many meshes you have
  • Keeps animations intact
  • Can be used while Animator performs animations
  • Perfect for combining just the colors of your model, no need for textures.
  • It is optimized and fastest merge method.
  • Does not generate too much memory garbage.
  • It is the simplest method of merge
  • Can be called more often than the "All In One" method
  • Fully compatible with mipmaps system
  • You do not need "Read/Write enabled" permissions in the textures import settings
  • It is compatible with all types of meshes!
Cons Of "Just Material Colors" Merge Method
  • This merge method works only with material colors. Any kind of texture is simply ignored.
  • It does not support special effects like Normal Maps and the like.
How this method works (Only Anima2D Meshes Merge Method)
This merge method is unique to the Skinned Meshes generated by the Unity Technologies Anima2D tool, and only works when the tool is also installed in your project. The merge process is similar to "All In One", but only works with Anima2D (Sprite Mesh Instance) meshes.

First, it combines all sprite meshes and sprites in just one mesh and one atlas.

After this, the Skinned Mesh Combiner adds a Combined Mesh Manager script to render atlas in merged mesh. This script is responsible for collecting the sprites, creating a new material, and applying the rendering data to the merged mesh, repeating the process with each frame.

Pros Of "Only Anima2D Meshes" Merge Method
  • Most of the time, it dramatically reduces the amount of draw calls
  • Always reduce calls to 1 draw call, regardless of how many meshes you have
  • Keeps animations intact
  • Can be used while Animator performs animations
  • Perfect for combining Sprite Mesh Instances from your character, no need for textures.
  • It is optimized and fastest merge method.
  • Does not generate too much memory garbage.
Cons Of "Only Anima2D Meshes" Merge Method
  • Does not match Sprite Mesh Instance colors.
  • You need to enable the "Read/Write enabled" option for all model textures in the import settings.
Problems with the drawing order of the Combined Sprites Meshes
Sometimes, by combining the sprites of your characters, you may notice that some sprites are over other sprites, an unwanted effect. For example, an arm may lie on top of the body of the character (which should be behind the body).

See the example below.

In the example, the body remained on the left arm and the neck remained on the face. That was not the expected result.
To correct this, we must change the GameObjects order of each part of the character in the hierarchy and this must be done before the merge begins. So we can organize which sprites will be on top of others. Knowing this need, the Skinned Mesh Combiner has a tool that provides you with an intuitive and easy to understand interface, so you can organize the rendering order of your character's meshes, all quickly and in a functional way! After organizing your character's meshes with this tool, the same rederization order you defined will be used, both after merging, as well as before doing it!

In the Skinned Mesh Combiner component, you need to be using the "Only Anima2D Meshes" blending method. So if the Skinned Mesh Combiner detects meshes to be combined, it will show you the "Set Correct Sprite Meshes Rendering Order" button. Click this button and the mesh rendering order organization interface will appear, so just use the / \ and \ / buttons to organize the mesh order. The higher a mesh is at the top, the higher it will be rendered over the others.

Warning About Anima2D
Anima2D is a tool developed and maintained by Unity Technologies, and like any other tool, it is also subject to bugs and flaws. Sometimes, after combining the meshes, you may notice some bugs in the meshes, but in the vast majority of cases the problem is caused by a flaw in the Anima2D scripts. Usually, recreating your character's bone structure or merging again can solve this problem. Usually the most common bugs are related to the rendering order of the Sprite Meshes after combined. You can always use the tool mentioned in the previous topic. If you need help, just contact mtassets@windsoft.xyz!
Undoing the Merge
When you undo the merged meshes that were previously generated are deleted from the scene, and the old meshes are re-activated. It's that simple! This is all done automatically. Never attempt to manually delete the meshes after a successful merge, because when you undo the merge, the Skinned Mesh Combiner will undo all that has been done and restore the meshes correctly.

Using additional effects like Normal Maps, Height Maps, and more
One Mesh Per Material: The "One Mesh Per Material" method can merge your meshes with special effects automatically without any problem, plus you will not have to configure anything other than your own shaders.

Just Material Colors: The "Just Material Colors" method is designed to support only material colors, so it does not support texture merging or special effects.

Only Anima2D Meshes: The "Just Material Colors" method is designed to support only material colors, so it does not support texture blending or special effects.

All In One: "All In One" supports special effects of your materials! You just need to enable support and configure it. After that, just merge! See below how to do the procedure.

Step 1 - First, activate support for the effect you want.

When you activate support for an effect, the Skinned Mesh Combiner, in addition to creating an atlas for your model's textures, it will also create an atlas with the textures of the effect you want. This atlas will be based on the texture atlas of your model.

Presentation
Step 2 - Select the property of your model materials, which store the texture / map of the desired effect! This is how the Skinned Mesh Combiner will know where to look for the textures of the effect that you also want to apply, to combine them!

The properties of the materials are like "variables" that store a texture. In the example above, the "_BumpMap" property (named Normal Map in Standard material Inspector) stores the normal map textures.

Skinned Mesh Combiner will scan the properties of all materials found in your model, and will allow you to select the property you want Skinned Mesh Combiner to find the textures/maps of the desired effect. That way you won't get lost! By default, Skinned Mesh Combiner will already show you the default property that stores such an effect.
Step 3 - Select the material property you provided (Material To Use) to store the atlas of the desired effect. In this case, the "_BumpMap" property of the material I supplied (my material is a Standard) will be responsible for storing the normal maps atlas that the Skinned Mesh Combiner generates.

Property Name To Find: It is nothing more than the name of the variable that stores the textures of the desired effect in its materials. For example, if you have enabled Normal Maps support, the Skinned Mesh Combiner will look for the variable "_BumpMap" in each shader of each material in your mesh, when the Skinned Mesh Combiner finds this variable it will extract the texture of normal map dest variable. When it has all the normal map textures, it will merge them into an atlas.

Property Name To Insert: It is the variable that will store the atlas texture of normal maps, in the final material. For example, at the end of the merge, the Skinned Mesh Combiner will look for the variable "_BumpMap" in the final material and will insert the normal maps atlas in this variable.
Tip to avoid differences in positioning of Normal Maps and other Maps in the mesh resulting from the merge
If you have textures and normal maps (or other types of maps like occlusion) and are trying to combine using the "All In One" method and your textures have different and varied formats, you can end up with situations where Normap Map for example, looks like be positioned incorrectly. As in the example below.

Note that after performing the merge, the normal map seemed more "simplified".
This happened because the textures have a lower resolution than the normal map textures. The textures are 256x256 while the normal maps are 512x512. So Normal maps end up being positioned only in the center of the texture and there are still some parts of Normal Maps left.

Due to the fact that Normal Map is larger than the texture itself, it ends up being poorly positioned over the texture.
The Skinned Mesh Combiner, upon identifying that this may occur, will display this warning to you.

Representation
To correct this problem simply click on the "Fix all textures where the resolution is different" button. Or if you prefer, do it manually, for that... Leave ALL the textures and maps of your model, in uniform resolutions. In my example model, I fixed this problem by simply resizing all textures, normal maps, occlusion maps and etc. to the uniform size of 512x512. So in the atlas, all textures will be the same size and the positioning will be perfect!

Edges Mip Map Size Tip
The edges in each texture of an atlas avoid distortions by mip maps, when the camera moves away from the 3D model, or remains at certain angles. The larger the border of each texture, the less chance of artifacts appearing. With SMC you have complete freedom to select the size of the borders you want to use in the textures of your atlases, and if you are noticing artifacts in your models, just increase the size of the borders. However, it is important that you never use borders larger than the textures themselves, because, in addition to increasing the time it takes to create the borders, you will also cause the texture to lose A LOT of quality. Monitoring all Resources (Like Meshes/Materials) found by SMC before merging
The Skinned Mesh Combiner has an area called "Debugging", available in the "Merge" tab. Debugging shows you all the features that Skinned Mesh Combiner found in your character, and all of these features will be combined, according to the blending method you have chosen. This even makes it easier for you to know ALL the textures that the Skinned Mesh Combiner has found and will match, as well as detected materials and meshes! All features shown in Debugging will be included in the merge, even if you combine in real time, or in the editor. Debugging does not look for resources or meshes in GameObjects marked to be ignored during the merge.

This tab displays all the meshes found. All meshes listed here will be combined. You can also click on the buttons to see which GameObject that mesh was found on. You can also see how many sub-meshes the mesh has, as well as the amount of vertices.

This tab displays all the materials found.

This tab shows all the textures found and if you chose the 'All In one' blending method, all these textures will be combined in atlases. You can see the resolution of each, type of compression, a preview of each, name and etc. The SMC also informs you if the texture is marked as "Read / Write Enabled" with the attribute "Readable"!
Some blending methods like "All In One" require that all of your textures are marked with "Read / Write Enabled" in the import settings for each texture. So SMC can read each one, copy its pixels and create atlases with it. If Skinned Mesh Combiner finds one or more textures without this option enabled, it will display a button for you to click and Skinned Mesh Combiner will activate this option quickly and automatically for you! You can also see which textures are readable for SMC, via the "Textures" tab in "Debugging".

How to Create Prefabs with SMC!
To create a prefab automatically after the merge, enable the "Save Prefab Of This" option in the Skinned Mesh Combiner component.

This is the recommended way.

Step 1 - Enable the "Save Data In Assets" option in the Skinned Mesh Combiner component.

Current limitations of the Skinned Mesh Combiner
The Skinned Mesh Combiner has some limitations. See below for what they are and their explanations.

Always ignore meshes that use Blendshapes - Knits that have blendshapes have a more complex weights and calculations for blendshapes to work well. In addition, if you or Animator need to modify the blendshapes at runtime, if the blendshapes meshes were disabled, it would cause visual problems. To avoid bugs and ease the operation of blendshapes, the Skinned Mesh Combiner will always ignore knits that use it.
Optimization tips for using Skinned Mesh Combiner
Avoid combining meshes multiple times in a short time
Doing this kind of thing can consume too much memory from the heap, and cause performance losses in your game. If your game has a customization system for example, just combine the meshes when the player finishes customizing.
Activate the Unity garbage collection in the Skinned Mesh Combiner settings.
Enabling this setting will cause the Skinned Mesh Combiner to clear the memory used by the data that was deleted when you undo the merge. This prevents your heap memory from filling up quickly. Use this option with caution as this can cause in-game freezes depending on how much data to exclude.
Combining several characters right at the beginning of the scene.
If you need to combine multiple characters (for example, NPCs) early in the game, it may be a bad idea to do this at run time, since too much memory can be consumed or cause freezes in the game. In these cases, give preference to merge through the Editor (Inspector) so that as soon as the scene is executed the characters will already be merged. If you prefer, you can enable the "Save Data In Assets" option. When you enable this option, whenever you merge meshes using the Inspector, data from those meshes will be saved to the files in your project. This makes it even easier to create pre-merged prefabs!
One Mesh Per Material or Just Material Colors to combine at runtime
The "One Mesh Per Material" and "Just Material Colors" merge method is faster and generates much less garbage besides being almost instantaneous, so it's a great merge method to use while your game runs. Even so, avoid merging many models at the same time, or many so-called merge in a short period of time.
Fixing Compile Error "The type or namespace name `Anima2D' could not be found. Are you missing an assembly reference?"
This error occurs when the Anima2D asset scripts and namespace are removed from the project. The Skinned Mesh Combiner can no longer detect the scripts, which causes compilation errors. The Skinned Mesh Combiner has a definition in your project settings, which indicates that there are Anima2D scripts in the project. To correct and eliminate any errors generated by the lack of Anima2D in your project, go to Player Settings > Other Settings and remove the text "MTAssets_Anima2D_Available" from the "Scripting Define Symbols" field. After removing this setting, the Skinned Mesh Combiner will know that there are no more scripts in the Anima2D tool and no more errors will be generated.

This field is available in "Player Settings> Other Settings"
Fixing Runtime Merge Error "Cannot combine mesh that does not allow access: MeshName..."
This error appears when you try to combine fabrics at run time, but the fabrics are not marked "Read/Write Enabled" in the import settings. To correct this error, activate the option "Read/Write Enabled" in your meshes and then click "Apply". After that it will be possible to read these meshes and combine them at run time!

Why isn't my atlas completely filled with my character's textures?
It is not always possible for textures to completely fill the atlas, even if they are perfect squares. Textures must have edges (as if they were tiled) in order to be able to support mipmaps with atlas. These borders increase the size of the textures and often make them no longer perfect squares. Another factor is the varying size of the textures or even the spacing between each texture in the atlas (which can be configured in the Skinned Mesh Combiner). But these are some of the factors that make it not always possible for textures to completely fill the atlas. See an example below.

Why do the tiled textures look stretched or look strange after merge?
See below for a good example of a tiled texture that looks stretched or strange after blending.

Representation
We first need to understand why this happens, and for that, we need to know how the options for blending tiled textures work. Note that this only happens in "All In One" blending mode. Let's talk about the "Merge Tiled Textures" option.

Skip All - The Skinned Mesh Combiner will simply skip any meshes that have tiled textures and exclude them from the merge.

Legacy Mode - The Skinned Mesh Combiner will map the UV of tiled textures as if they were any other normal textures, this gives the impression that the texture is stretched on the mesh, as it is no longer tiled and in fact stretched to fit across the entire UV of the mesh .

Improved Mode - The Skinned Mesh Combiner uses a new algorithm that is capable of supporting tiled textures, even if the UV coordinates of the mesh vertices are negative. The thing here, however, is that the Skinned Mesh Combiner will tile these textures within the final atlas. If a mesh has a very large tile, the texture will be repeated more and more times within the atlas, which will end up generating a low quality when the texture is rendered on the mesh.

Look at the image above. Note that in the first part the original shape of the texture is displayed. In the second part the texture seemed stretched because the tiled textures blend mode used was "Legacy Mode". In the third part, the texture seemed strange because the blend mode used was "Improved Mode" and the mesh of the skirt has a huge tile.
See below, in the blender, the UV mapping of the character's skirt mesh.

Note that the skirt's UV mapping is much larger than the texture actually is, which generates a large tiling. So, with "Improved Mode" the texture has lost a lot of definition. It was repeated over and over again to the point that there was not so much room to keep repeating while maintaining the definition of the texture.
Now let's make a comparison, with a tiled mesh, but without a huge UV map, so that it is not repeated over and over again in the final atlas.

After combining my mesh using Improved Mode, a part of my character has the texture in the wrong place. Why does it happen?
This is not common. But there is a minuscule chance of this happening, depending on how your model had the UV mapped. See the example below to understand better.

In the example blend, the UV mapping of the character's eye was in the wrong place. The texture on the left is the original texture of the character's eye. The texture on the right is the final atlas generated after blending. Note that the texture of the eye in the atlas, had to be partially repeated, because some mesh that also uses this texture, has a tiling.
Explaining what happened: There is some sub-mesh of this character, which uses the same material (and consequently the same texture) as the eye. This sub-mesh has a texture tile, which caused the texture to be partially repeated in the atlas, and like the texture of the eye, it is together with that circular and white texture, the texture of the eye ended up being moved too.

This was because of the way this character was mapped, perhaps a somewhat incorrect form for merges. Unfortunately, it is not possible to predict that this would happen through an algorithm, so the Skinned Mesh Combiner was unable to apply a correction for this. Everything else in the character model was correctly mapped. The eye was only mapped incorrectly because it was in a texture that needed to be repeated, thus, shifting its position and leaving the position to which it should be.

To correct this, first, we can simply use a program like Blender, to separate the vertices of our character's eye, as in the figure ...

Warning before using SMC with HDRP, URP or different standard shaders
URP and HDRP (rendering pipelines) have new default property names in the shaders. For example, in the embedded pipeline (Unity default) the default name of the property that stores your character's textures is "_MainTex", however, in HDRP and URP this default name has been changed to "_BaseMap". Therefore, if you use HDRP, URP shaders or different shaders (created by third parties for example), you can change the properties where the SMC seeks the textures. To do this, deselect the option "Default Main Tex. Prop.", Doing so the Skinned Mesh Combiner will allow you to change the default properties where the textures will be searched and applied.

About This Component: Skinned Meshes Bones Manager
Sometimes we need to debug our 3D character models, for example, we need to see which vertices are controlled by each of our character's bones. Sometimes we need to see all the bones that are part of our character's bone hierarchy, and consequently which are the bones (Transform) that move and animate our character. When creating a character customization system, where each outfit is saved with a different FBX file, we need to make the clothes animated using the character's bone hierarchy of bones, to ensure perfect animations. The "Skinned Mesh Bones Manager" component provided by the Skinned Mesh Renderer can help you with all of this, both in the Editor and at run time. Keep reading to understand more about everything!
Visualizing The Hierarchy of Bones That Animates a Mesh
The Skinned Meshes Bones Manager allows you to see the entire bone hierarchy that currently animates and deforms the mesh it is in now. You can see the name of all the bones, the index of each one and other data.

The component. The "Shirt" mesh is influenced by a bone hierarchy that contains a total of 18 bones.

We can see the name of each bone, the amount of vertices that each bone deforms, in addition to the index pertaining to each bone, in the bone hierarchy. Visualizing All Vertices Influenceds By a Bone
By clicking on the "Vertices" button for a given bone, Skinned Meshes Bones Manager will display an interface on the scene, that interface will display points where each vertex controlled by the respective bone is. The color of the point indicates how much weight the bone exerts at that vertex. The greener the vertex is, the smaller the weight, the redder the vertex, the greater the weight. When you click on a point (vertex), more details are displayed.

In the image above, the "ColumnU" bone of the "Shirt" mesh was selected. The editor reported that this bone influences 619 vertices and renders all vertices influenced by the bone. Thus, you know where and at what points this bone controls the "Shirt" mesh. It's great for debugging! There are still more bones influencing different points of the "Shirt" mesh, which can be seen in the component.

The component only displays the influenced vertices, of the mesh to which it is located. That is, it shows all the bones in the bone hierarchy that animates the "Shirt" mesh, but only shows the influenced vertices of the "Shirt" mesh. If you want to see the influenced vertices, from the "Backhair" mesh for example, you need to add the "Skinned Mesh Bones Manager" component to the "Backhair" mesh, and the component will display which vertices of the "Backhair" mesh are affected by the different bones.
Making a Mesh Use The Bone Hierarchy of Another Mesh, To Be Animated. Customizable Character, Several FBX Files, One For Each Outfit.
You may have already come across a character customization system. Typically, many games use multiple FBX files, a different file for each mesh. In the following example character, the head is the main mesh, has its hierarchy of bones and the clothes must be attached to it to form the body.

It seems that the character has all the meshes together, but it doesn't. Each character's outfit is a different mesh, a different FBX file.
However, note that when I move the bones of the head bone hierarchy, the rest of the body does not move, even if I move bones in the spine, arm, etc. This is because each mesh (clothing) of the character, still has its own hierarchy of bones, and if I want to make the animations of the shirt for example, I need to move the bones of the shirt, not move the bones of the head. We are going to use the Skinned Mesh Bones Manager to make all the clothes come alive using the head bone hierarchy, so when we move any bone from the head bone hierarchy, the clothes will also be deformed. Keep reading to learn more!

Demonstration.
First, in order for it to be possible for a mesh to be animated using the bone hierarchy of another mesh, both must have a hierarchy of identical bones. See the example below, where I show the hierarchy of bones of the head, and the shirt.

The two meshes have an IDENTICAL bone hierarchy, with the same names, bones in the same indices, etc. To make this possible, when creating my model, I did the following: I created your body, created and placed the bone hierarchy and assigns the weights making the bone hierarchy animate my character's body. So, I created 5 copies of my character's body. In the first copy, I deleted all the vertices of my character's body, leaving only the head. In the second copy I deleted all the vertices of my character, leaving only the vertices pertaining to the shirt, and so on for the remaining 3 copies. In the end, the head, shirt, pants, hair and shoes were left. All using the same bone hierarchy.

All bones in the bone hierarchy must remain there, in all meshes, even if not all bones are used. For example, in the head, the leg bones, arms, etc. still remain in the hierarchy, but they do not move anything, only the bones of the neck and head make the movement of the head. In the T-shirt, the bones of the legs, head and etc. are still there, but only the bones of the spine, arms and etc. can animate the head.

Converting the combined mesh to a static mesh
If the meshes are combined, it is possible to convert the mesh resulting from the merge, into a static mesh. That is, it will cease to be a Skinned mesh and become a common mesh (Mesh Renderer + Mesh Filter). The common mesh will have the vertices at the position of the vertices of the Skinned mesh, the moment this method is called. After converting the Skinned mesh to static, the Skinned mesh will no longer be rendered, only its static version.
Script C#


using UnityEngine;
using MTAssets.SkinnedMeshCombiner;
    
public class Test : MonoBehaviour {
    
    public SkinnedMeshCombiner meshCombiner;

        //Convert the mesh result of merge, to static
        meshCombiner.ConvertCombinedMeshToStaticMesh();
    }
}
